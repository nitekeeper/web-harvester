How should click-to-edit be wired in?

Three approaches for implementing inline renaming in the destination list

A

Edit state in DestinationRow Recommended

DestinationRow holds isEditing + draft local state. Clicking the label flips isEditing on; Enter or blur saves and exits. NameStack receives isEditing + callbacks and renders either a <span> or an <input>.

Pros

  • Row owns state, NameStack stays display-only — matches existing pattern
  • No new files needed
  • Minimal prop changes

Cons

  • DestinationRow grows slightly
B

Edit state inside NameStack

NameStack receives the id and onRename handler directly and owns its own edit state internally. Fewer props thread through DestinationRow.

Pros

  • Less prop threading through DestinationRow

Cons

  • NameStack becomes business-logic-aware — breaks the display-only pattern
  • Harder to test NameStack in isolation
C

New EditableLabel component

Extract a reusable EditableLabel component (click-to-edit generic widget) and use it inside NameStack.

Pros

  • Reusable elsewhere if needed
  • Clean abstraction

Cons

  • Over-engineering — only one use case right now
  • New file for a small feature
